## 2. 경영진 보고

### 2.1 이번 세션의 성과 요약

본 세션을 통해 ERP 프로세스 학습 시뮬레이터는 **단일 목적의 교육 데모에서, 교육과 회계감사 대응을 함께 지원하는 업무 자산으로 전환**되었습니다. 총 16개 이슈를 처리했으며, 코드 변경은 12건의 커밋으로 반영되어 모두 운영 환경에 배포되었습니다.

성과는 세 갈래로 요약됩니다. 첫째, **초보자 진입장벽을 제거했습니다.** 기존에는 접속 즉시 잠긴 메뉴와 PIN 입력란만 노출되어 이 도구가 무엇인지 설명하는 화면이 전혀 없었고, 화면마다 10개 내외의 버튼이 동시에 표시되어 순서를 알기 어려웠습니다. 이제는 학습 성과와 커리큘럼을 소개하는 랜딩 페이지가 먼저 나오고, 각 화면은 "지금 할 일" 하나만 크게 보여준 뒤 나머지는 잠금 처리하여 순차적으로 열립니다.

둘째, **30분 단일 스토리 발표 체계를 갖추었습니다.** 9개 챕터에 정확히 30분을 배분하고, 챕터마다 전면 스토리 슬라이드가 팝업된 뒤 실제 화면 시연으로 이어지는 리듬을 구현했습니다. 발표자는 경과 타이머와 챕터별 발표 포인트를 보며 진행할 수 있어, 별도 슬라이드 자료 없이 이 도구 하나로 발표가 가능합니다.

셋째, 그리고 가장 중요하게는 **회계감사 대응 자료로서의 신뢰성을 확보했습니다.** 외부 벤치마킹에 앞서 수행한 코드 점검에서, 감사인에게 제시할 분개원장이 시뮬레이션 변수와 연결되어 있지 않아 **화면과 다른 금액을 출력할 수 있는 결함**을 발견했습니다. 이는 감사 자료로서는 치명적인 문제였으며, 랜딩 페이지 신설보다 우선하여 해결했습니다. 이후 전표번호 체계, 선행문서 추적, 통제점 명시, 통제 매트릭스, 감사 패키지 인쇄, 재현용 스냅샷까지 감사 문서화의 표준 구성요소를 단계적으로 반영했습니다.

### 2.2 발견된 중대 결함과 조치

본 세션의 가장 큰 실질적 가치는 **선제적 코드 점검을 통한 결함 발견**에 있습니다. 사용자가 "감사 대응 자료로 쓰겠다"는 목적을 제시한 시점에, 외부 리서치와 병행하여 기존 코드를 점검한 결과 세 건의 결함을 확인했습니다.

가장 심각한 것은 분개원장 화면이 시뮬레이션 변수를 전달받지 못한 채 **고정된 금액을 코드에 직접 기재**하고 있었다는 점입니다. 사용자가 변수 입력 화면에서 발주 단가를 변경하면 각 업무 탭은 즉시 새 금액을 반영하지만, 정작 인쇄하여 감사인에게 전달할 분개원장은 최초의 숫자를 그대로 유지했습니다. 감사 자료가 화면과 모순되는 상태였던 것입니다. 이 밖에 직전 세션에 신설한 일반경비 전표가 원장에 누락되어 있었고, 인쇄 기능은 있으나 인쇄 전용 서식이 없어 내비게이션과 사이드바가 함께 출력되는 문제가 있었습니다.

세 건 모두 수정 후 실제 브라우저에서 검증했습니다. 검증 방법으로 발주 단가를 5.00달러에서 4.00달러로 변경했고, 분개원장의 미착 계정이 500달러에서 400달러로, 구매가격차이가 50달러에서 150달러로 즉시 연동되는 것을 확인했습니다.

### 2.3 투자 대비 효과

본 세션의 산출물은 **별도 라이선스 비용이나 서버 운영비 없이** 기존 자산의 활용 범위를 확장했습니다. 산출물 전체가 단일 HTML 파일이며 정적 호스팅으로 운영되므로 추가 인프라 투자가 발생하지 않습니다.

교육 측면에서는 신입 및 현업 대상 ERP 프로세스 교육을 외부 과정 없이 사내에서 진행할 수 있습니다. 감사 측면에서는 내부감사 또는 외부감사 질의 시 "우리 원가 프로세스는 이렇게 흐른다"를 설명하는 표준 자료로 활용할 수 있으며, 특히 감사인이 가장 먼저 묻는 **표준원가와 실제원가의 차이를 재고와 매출원가 중 어디로 배부했는가**라는 질문에 화면 토글 하나로 두 정책의 손익 영향을 대비하여 제시할 수 있습니다.

### 2.4 잔여 리스크 및 권고

현재 `main`과 `workspace` 두 브랜치가 공통 이력 없이 분리 운영되고 있습니다. 당장의 운영에는 지장이 없으나, 향후 저장소를 정리하실 계획이라면 두 구조를 어떻게 통합할지 별도 검토가 필요합니다.

또한 본 시뮬레이터는 가상 회사와 가상 거래로 구성된 교육용 도구입니다. 감사 대응에 활용할 때는 **실제 회계기록이 아니라 프로세스와 통제 구조를 설명하기 위한 참고자료**임을 명확히 해야 하며, 이를 위해 인쇄되는 모든 페이지에 해당 취지의 문구가 자동으로 삽입되도록 처리했습니다.

## 3. 개발자 보고

### 3.1 아키텍처 제약과 그 영향

본 산출물은 React 18과 Tailwind CSS를 CDN으로 로드하고 브라우저에서 Babel이 JSX를 변환하는 **빌드리스 단일 HTML 파일**입니다. 파일 크기는 약 386KB, 약 5,300줄 규모입니다. 이 구조는 배포가 단순하고 의존성 관리 부담이 없다는 장점이 있는 반면, 모듈 분리가 불가능하여 모든 컴포넌트가 하나의 스크립트 블록에 존재합니다. 따라서 이번 세션의 모든 변경은 동일 파일 내 국소 수정으로 이루어졌으며, 컴포넌트 간 결합도를 낮추기 위해 **공용 상수와 파생 함수를 단일 진실 공급원(single source of truth)으로 두는 방식**을 일관되게 적용했습니다.

PIN 값은 소스에 `__APP_PIN__` 플레이스홀더로 존재하며 Vercel 빌드 시 `build.js`가 환경변수로 치환합니다. 로컬 검증 시에는 임시로 값을 넣고 검증 후 반드시 플레이스홀더로 복원하는 절차를 매 작업마다 수행했습니다.

### 3.2 핵심 리팩터링 — 분개원장의 파생화

이번 세션의 기술적 중심은 분개원장을 **파생 데이터로 재작성**한 것입니다.

변경 전 `LedgerTab`은 `simParams`를 프로퍼티로 받지 않은 채 `if (mmSteps.migo_gr) entries.push({ ... amt: 500 ... })` 형태로 금액을 직접 기재했습니다. 반면 각 업무 탭은 `buildJournals(simParams, materials, lang)`가 반환한 값을 사용했습니다. 동일한 사실에 대해 두 개의 계산 경로가 존재했고, 그중 하나가 입력 변화에 반응하지 않는 구조였습니다.

변경 후에는 `LEDGER_DOCS`라는 명시적 문서 매핑 배열을 도입하여, 실행된 단계 키를 대응하는 저널 키로 연결하고 `buildJournals`의 출력에서 차변·대변 라인을 추출하는 방식으로 통일했습니다. 부수 효과로 원장 설명 문구가 언어 토글을 따르게 되었습니다. 기존에는 설명이 한국어로 하드코딩되어 영문 모드에서도 한국어가 표시되고 있었습니다.

계정별 잔액 패널 역시 고정값 7행이었으나, 전기된 분개 라인을 계정 기준으로 집계하는 **시산표**로 교체했습니다. 계정 문자열에 부가 설명이 붙어 있어(`원재료 RM001 (재고)`) 괄호 이하를 제거하는 정규화 함수를 두어 동일 계정이 한 행으로 합산되도록 했습니다. 차변 합계와 대변 합계의 일치 여부를 배지로 표시하여 복식부기 정합성이 화면에서 즉시 확인됩니다.

이후 감사 패키지 기능을 추가하면서 이 계산을 `buildLedger()` 함수로 추출하여 화면 탭과 인쇄 패키지가 동일한 계산을 공유하도록 했습니다. 두 산출물이 서로 다른 숫자를 낼 가능성을 구조적으로 차단한 것입니다.

### 3.3 점진적 노출(Progressive Disclosure) 구현

`StepButton` 컴포넌트를 세 가지 상태로 분기하도록 재작성했습니다. 완료된 단계는 체크 표시가 붙은 한 줄 요약으로 축약되고, 다음에 실행할 단계는 전체 폭 CTA 버튼으로 강조되며, 아직 열리지 않은 단계는 자물쇠 아이콘의 흐린 한 줄로 표시됩니다. 다음 실행 단계는 `steps` 객체의 선언 순서에서 첫 번째 미완료 키를 찾는 방식으로 판정하므로, **호출부를 전혀 수정하지 않고** 모든 탭에 동시 적용되었습니다.

부가 시나리오(반품, 재고실사, 후속차변, 고객반품, 자재재평가)는 `AdvancedSection` 컴포넌트로 감싸 기본 접힘 처리했습니다. 그 결과 생산 탭 기준으로 기존에 10개 버튼이 동시 노출되던 화면이 **CTA 1개와 잠금 9줄**로 정리되었습니다.

### 3.4 감사 기능의 데이터 설계

감사 관련 기능은 세 개의 모듈 수준 상수를 단일 공급원으로 설계했습니다.

`LEDGER_DOCS`는 문서 순번과 선행 문서 관계를 담습니다. 각 항목에 중립 접두사(`PO`, `MD`, `AC`, `PR`, `SO`, `DL`, `IV`, `CC`, `CK`, `ML`, `PA`)와 선행 단계 키를 부여하여, 실행 시점에 순번을 매기고 선행 문서 번호를 해석합니다. SAP 실제 번호대역을 사용하지 않은 것은 기존 디브랜딩 정책과의 일관성 때문입니다.

`CONTROLS`는 10개 통제점을 리스크, 통제 기술, 예방·적발 구분, 빈도, 방법, 재무제표 주장 컬럼과 함께 정의합니다. 이 배열 하나가 **단계 버튼의 방패 배지**, **통제 매트릭스 탭**, **워크스루 시트의 통제 열**, **인쇄 패키지의 통제 매트릭스** 네 곳에 동시 공급됩니다.

`GLOSSARY`는 14개 약어 정의를 담아 용어집 모달과 인쇄 패키지에 공급됩니다.

인쇄 패키지는 `body.print-pack` 클래스와 `@media print` 규칙을 조합하여, 인쇄 시에만 앱 본체를 숨기고 감사 문서를 노출하는 방식으로 구현했습니다. 화면 표시에는 전혀 영향을 주지 않으며 외부 PDF 라이브러리를 추가하지 않았습니다.

### 3.5 검증 방법

모든 변경은 로컬 정적 서버에서 실제 브라우저를 구동하여 검증했습니다. 콘솔 오류 확인, DOM 상태 조회, 단계 실행 후 상태 전이 확인을 반복했으며, 특히 다음 항목은 **결과를 수치로 확인**했습니다.

- 발주 단가 변경 시 원장 연동: GR/IR 500달러 → 400달러, 구매가격차이 50달러 → 150달러
- 시산표 차변·대변 일치 배지 표시
- 문서 추적 체인: `PO-0001 → MD-0001 → AC-0001 → AC-0002`
- 스냅샷 왕복: 단계 추가와 단가 변경(3.75달러)이 복원 후 정확히 재현
- 발표 모드 9개 챕터 순회 및 마지막 챕터에서 다음 버튼 비활성화

검증 과정에서 브라우저 프리뷰 도구가 수차례 응답 지연을 일으켰으나, 정적 서버는 정상 응답(HTTP 200, 386KB)했고 도구 재시작 후 동일 코드가 정상 동작함을 확인하여 **애플리케이션 결함이 아닌 자동화 도구 측 문제**로 판정했습니다.
